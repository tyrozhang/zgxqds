# /zreq - 需求澄清助手

专为系统化需求审查设计，通过结构化问答将模糊概念转化为明确规范。

## 调用方式
用户输入：/zreq [可选：需求文档路径]

## 执行步骤

### 第 1 步：定位需求文档

读取需求文档，按优先级查找：
1. 用户指定的路径
2. ./docs/requirement.md
3. ./docs/requirements.md
4. ./requirement.md
5. ./requirements.md
6. ./README.md

如果找不到，询问用户："请提供需求文档的路径"

### 第 2 步：识别模糊点

扫描文档，找出需要决策的内容：

**标记符号：**
- `[TBD]`、`[待确定]`、`[TODO]`、`[?]`
- `[XXX]`、`[FILL]`、`[补充]`

**模糊词汇：**
- "可能"、"或许"、"考虑"、"待定"
- "若干"、"一些"、"多个"、"适量"
- "快速"、"高效"、"友好"（无具体指标）
- "支持"（无具体范围）、"可选"（无默认行为）

**不一致信号：**
- 同一功能在不同章节描述不同
- 前后文矛盾的需求
- 缺少边界条件（如"最大"、"最小"限制）

### 第 3 步：呈现决策清单

以编号清单形式向用户展示，每项包含：

```
需要您决策的问题（共 {N} 项）：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【{序号}】{问题类别}：{问题简述}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 位置：{章节/段落}
📝 当前描述："{原文摘录}"

❓ 需要明确：{具体问题}

💡 建议选项：
   A. {选项A}
   B. {选项B}
   C. {其他，请说明}

请回复您的决策（如：选A / 自定义答案）
```

**等待用户逐个确认**，每确认一个立即执行第 4 步。

### 第 4 步：即时更新文档

用户确认决策后，立即：

1. **替换模糊标记**
   - 将 `[TBD]` 替换为具体决策内容
   - 删除 `[待确定]` 等占位符

2. **添加决策注释（可选）**
   ```markdown
   <!-- 决策记录：{日期} - {决策内容} -->
   ```

3. **更新相关段落**
   - 确保决策在文档中一致体现
   - 修正因决策而产生的矛盾描述

### 第 5 步：维护决策记录

在文档末尾检查是否存在 `## 决策记录` 章节：

**如不存在，创建：**
```markdown
## 决策记录

| 日期 | 问题 | 决策 | 决策原因 |
|------|------|------|----------|
| {日期} | {问题简述} | {决策内容} | {用户提供的原因，如有} |
```

**如已存在，追加新行**

### 第 6 步：同步相关文档

检查并同步变更到以下文档（如存在）：

| 文档 | 同步内容 |
|------|----------|
| docs/design.md | 功能设计需与需求保持一致 |
| docs/plan.md | 实施计划需反映决策变更 |
| docs/api.md | API 设计需符合需求规范 |
| CHANGELOG.md | 记录需求变更历史 |

**同步前需用户确认：**
> "检测到相关文档 `{文件名}`，是否同步此决策？（是/否/先给我看差异）"

### 第 7 步：完成报告

所有决策处理完毕后，输出总结：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 需求澄清完成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📊 统计：
   - 识别模糊点：{N} 个
   - 已决策：{N} 个
   - 暂缓处理：{N} 个（如有）

📝 更新文档：
   - {主文档路径}
   - {同步的文档列表}

💾 决策记录已保存至：
   - {主文档} 的"决策记录"章节

⚠️ 注意事项（如有）：
   - {需要后续跟进的事项}
```

## 特殊处理规则

### 规则 1：范围确认
开始处理前，必须先确认：
> "我将分析 `{文档路径}` 中的需求。请问：
> 1. 只更新此文档，还是需要同步修改其他文件？
> 2. 是否有特定章节需要重点关注或跳过？"

### 规则 2：术语澄清
当遇到可能混淆的术语时（如"测试"vs"练习"），主动澄清：
> "您提到的'{术语}'，是指：
> A. {解释A}
> B. {解释B}
> 请确认，以确保我理解正确。"

### 规则 3：决策粒度控制
当用户给出过于笼统的决策时，追问细节：
> "您选择了'{决策}'，为准确更新文档，请进一步明确：
> - {具体问题 1}
> - {具体问题 2}"

### 规则 4：暂缓标记
对于暂时无法决策的项目：
- 保留 `[TBD]` 标记
- 在决策记录中标记为"待后续决策"
- 设置提醒条件（如"等待 UI 设计确定"）

## 边界情况处理

| 情况 | 处理方式 |
|------|----------|
| 未找到模糊点 | 报告："文档已足够明确，未发现 [TBD] 或模糊描述" |
| 用户中途退出 | 保存已确认的决策，标记未完成的项 |
| 决策导致新模糊点 | 立即提出，加入当前决策清单 |
| 文档格式混乱 | 先询问是否先整理格式，再开始澄清 |
| 决策与现有内容矛盾 | 指出矛盾，让用户决定优先级 |
